A Tecnologia por tras da Voz
Hardware, software, protocolos e comunicacao — como dois microcontroladores,

1. Visao Geral da Arquitetura

O GESTUUM e um sistema distribuido composto por 2 microcontroladores que se comunicam sem fio, um aplicativo web (PWA) para configuracao, e um pipeline de audio pre-gerado por inteligencia artificial. Nenhuma parte do sistema depende de internet para funcionar no dia a dia — tudo roda localmente, no pulso do usuario.

SENSOR B SENSOR A (mao direita) (mao esquerda) M5StickC Plus2 M5StickC Plus2 + HAT-SPK2 | | | ESP-NOW (canal 13) | | IMU 50Hz, <3ms | | --------------------------------> | | | | [CEREBRO] | Reconhecimento | Audio I2S + BT | BLE + Serial | SPIFFS (5.4 MB)

O Sensor A e o cerebro do sistema — faz o reconhecimento de gestos, toca audio, e se comunica com o celular. O Sensor B e um sensor puro que envia dados IMU 50 vezes por segundo.

2. Hardware: Os Dispositivos

2.1 Sensor A — O Cerebro (mao esquerda)

Processador

MCU: ESP32-PICO-V3-02 (dual-core Xtensa LX6, 240 MHz)

RAM: 520 KB SRAM + PSRAM externo

Flash: 8 MB (2.5 MB firmware + 5.4 MB SPIFFS)

Placa: M5StickC Plus2

Sensores e Perifericos

IMU: MPU6886 (acelerometro + giroscopio, 6 eixos)

Tela: TFT 135x240 px (rotacao: 240x135 landscape)

Audio: HAT-SPK2 (MAX98357A, I2S, 1W)

Bateria: LiPo 200 mAh integrada

Funcoes: Captura IMU local, recebe IMU do Sensor B, reconhecimento de gestos (Matrix3D + DTW + Orbital), reproducao de audio (I2S + Bluetooth A2DP), servidor BLE GATT para configuracao via celular, interface serial USB para computador.

O conceito do GESTUUM e sempre com dois sensores: mao esquerda (contexto) + mao direita (objeto) = frase composta. Os dois dispositivos juntos formam a unidade funcional do sistema.

2.2 Sensor B — O Sensor Puro (mao direita)

Hardware

MCU: ESP32-PICO-V3-02 (identico ao Sensor A)

Placa: M5StickC Plus2

IMU: MPU6886 (mesma configuracao)

Tela: TFT 135x240 (status + bateria)

Funcao

Unica tarefa: Ler IMU a 50 Hz e enviar via ESP-NOW

Pacote: 20 bytes (timestamp + 6 eixos + bateria)

Sequencia: Numero sequencial para detectar perda de pacotes

Reconexao: Heartbeat broadcast a cada 10s se desconectar

O Sensor B nao faz nenhum processamento de gestos — ele e um streaming puro de dados IMU. Toda a inteligencia esta no Sensor A. Isso simplifica o firmware e reduz consumo de energia.

3.1 ESP-NOW — Comunicacao entre Sensores

O ESP-NOW e um protocolo proprietario da Espressif (fabricante do ESP32) que permite comunicacao ponto-a-ponto sem WiFi. Nao precisa de roteador, nao precisa de internet, nao precisa de pareamento complexo.

Por que ESP-NOW e nao WiFi ou Bluetooth?
  • Latencia: ESP-NOW < 3 ms vs WiFi ~50-200 ms vs BLE ~50 ms
  • Overhead: Pacotes de 20 bytes sem stack TCP/IP (zero handshake)
  • Energia: Transmissao curta e direta, sem manter conexao ativa
  • Simplicidade: Um dispositivo envia, outro recebe — sem servidor, sem IP

Para dados de IMU a 50 Hz (um pacote a cada 20 ms), cada milissegundo importa. O ESP-NOW e a unica opcao viavel nessa faixa.

Configuracao tecnica:

  • Canal: 13 (frequencia 2.472 GHz) — evita canais WiFi comuns (1, 6, 11)
  • Heartbeat: Pacote de 3 bytes a cada 1000 ms para detectar desconexao
  • Timeout: 5000 ms sem heartbeat = dispositivo desconectado
  • Pareamento: Sensor B descobre Sensor A via broadcast de heartbeat

3.2 Pacotes ESP-NOW — O Que Trafega no Ar

Cada tipo de mensagem tem um formato especifico, otimizado para o minimo de bytes:

PacoteTipoTamanhoDirecaoConteudo
IMUPacket0x0120 bytesB → Atimestamp, seq, ax/ay/az, gx/gy/gz, bateria
EmergencyPacket0x044 bytesBroadcastnivel (normal/urgente/critico)
HeartbeatPacket0x053 bytesBidirecionalremetente (A/B), bateria (%)
SetChannelPacket0x072 bytesA → todosnovo canal WiFi
O pacote IMU e o mais frequente: 50 pacotes por segundo, 20 bytes cada = 1000 bytes/segundo. Em uma hora de uso, isso e apenas ~3.5 MB de dados trafegando pelo ar — extremamente eficiente.

3.3 Pareamento Automatico

Quando os dispositivos sao ligados, o pareamento acontece automaticamente:

1. Sensor A liga → entra em modo PAIRING → broadcast heartbeat 2. Sensor B liga → escuta heartbeat → envia heartbeat de resposta 3. Sensor A recebe → flag marcada no ISR (interrupcao) 4. No loop() principal → esp_now_add_peer() com MAC do Sensor B 5. Pareamento completo → ambos entram em modo IDLE Tempo tipico: < 2 segundos apos ligar os dois

Seguranca da implementacao: O pareamento usa callback de interrupcao (ISR) que seta apenas um flag — o processamento pesado (adicionar peer) e feito no loop principal. Isso evita crash por chamada de funcao bloqueante dentro da ISR.

3.4 BLE GATT — Configuracao pelo Celular

O Bluetooth Low Energy e usado para configuracao — o usuario conecta pelo celular (via web app) e pode treinar gestos, trocar voz, mudar idioma, ajustar sensibilidade.

Servico BLE (GATT): UUID: 4e5f6a7b-8c9d-0e1f-2a3b-4c5d6e7f8a9b Caracteristicas: ├─ CMD_TX (Write) ...8a9c → App envia comandos JSON ├─ CMD_RX (Notify) ...8a9d → Dispositivo responde JSON └─ AUDIO_TX (Write) ...8a9e → Upload de audio (base64)

Protocolo: Comandos em JSON delimitados por newline (\n). Buffer de 512 bytes. Chunking de 20 bytes com 15 ms de delay entre fragmentos. Reconexao com backoff exponencial (2s, 4s, 8s).

Ativacao: BLE fica desligado por padrao (economia de energia). Ativado por: segurar botoes A+B por 3 segundos, ou triplo-clique no botao B.

3.5 Bluetooth A2DP — Audio Externo

Alem do alto-falante interno (HAT-SPK2), o Sensor A pode transmitir audio para qualquer caixa de som Bluetooth via protocolo A2DP (Advanced Audio Distribution Profile) — o mesmo protocolo que celulares usam para fones de ouvido.

  • Formato: PCM 16-bit, 48 kHz, mono
  • Coexistencia: BLE (config) e A2DP (audio) funcionam simultaneamente no mesmo ESP32
  • Speaker salvo: MAC da caixa de som e gravado em NVS — reconecta automaticamente
  • Buffer: Mutex-protected com alocacao em PSRAM para audios grandes
O audio interno (HAT-SPK2) e suficiente para ambientes quietos. A caixa Bluetooth e recomendada para ambientes ruidosos, como escola ou hospital, onde o volume de 1W do speaker interno pode nao ser suficiente.

3.6 USB Serial — Computador

A conexao por cabo USB-C e usada para: instalacao inicial do firmware, debug durante desenvolvimento, e como alternativa ao BLE para configuracao.

  • Baud rate: 115200
  • Protocolo: JSON por linha (\n), buffer de 1024 bytes
  • Funciona em paralelo com BLE — ambos aceitam os mesmos comandos
  • API Web: Web Serial API (Chrome/Edge) — acesso direto do navegador ao USB

4.5 Deteccao de Stroke (Inicio e Fim do Gesto)

Um dos problemas mais dificeis e saber quando o gesto comeca e quando termina. O GESTUUM usa um sistema de tres fases:

FaseCondicaoDuracaoFuncao
CooldownApos double-tap15 amostras (300 ms)Ignorar vibracao do toque
CalibracaoMao parada5 amostras (100 ms)Medir magnitude de repouso
Onset3 amostras consecutivas > repouso + 0.3gVariavelInicio do movimento
GravacaoMovimento ativo1-3 segundos (50-150 amostras)Captura dos dados
Offset25 amostras consecutivas < repouso + 0.15g500 ms de quietudeFim do movimento
O threshold de offset (0.15g) e mais baixo que o de onset (0.3g) propositalmente — e mais dificil "sair" de um gesto do que "entrar". Isso evita falsos finais quando o usuario faz uma pausa no meio do movimento.

5.2 Sistema de Arquivos SPIFFS

O SPIFFS (SPI Flash File System) armazena todos os dados do usuario:

/ (raiz do SPIFFS — 5.44 MB) ├── a/ (audio — abreviado pelo limite de 31 caracteres) │ ├── h/ (homem — voz masculina adulta) │ │ ├── agua.wav │ │ ├── comida.wav │ │ ├── ctx_01_i_want.wav │ │ └── ... (~30 arquivos WAV) │ ├── m/ (mulher — voz feminina adulta) │ ├── n/ (menino — voz masculina infantil) │ └── i/ (menina — voz feminina infantil) │ └── g/ (gestos — banco de dados de templates) ├── gestures.json (banco de gestos treinados) ├── contexts.json (8 contextos: "I want", "I need"...) └── custom.json (gestos customizados pelo usuario)
Limitacao: O SPIFFS so suporta UMA voz por vez no dispositivo (5.44 MB ÷ ~30 arquivos WAV = ajuste apertado). Trocar de voz exige re-upload via web app. Backup obrigatorio antes de trocar.

5.3 NVS — Configuracoes Persistentes

O NVS (Non-Volatile Storage) guarda configuracoes que sobrevivem a reinicializacao:

  • Preferencia de voz: homem/mulher/menino/menina (1 byte)
  • Nome do dispositivo: nome BLE customizado (max 24 caracteres)
  • MAC do speaker BT: endereco da caixa de som salva (6 bytes)
  • Canal ESP-NOW: canal WiFi atual (1 byte)
  • Nivel de gesto: limitado/padrao/avancado

7.3 APIs Web Utilizadas

Web Serial API (navigator.serial)
  • Permite o navegador acessar a porta USB diretamente — sem driver, sem terminal
  • Disponivel apenas no Chrome e Edge (nao funciona no Firefox ou Safari)
  • Usado para: instalacao de firmware, configuracao rapida, debug
Web Bluetooth API (navigator.bluetooth)
  • Permite o navegador conectar via Bluetooth Low Energy
  • Requer gesto do usuario (clique) para iniciar pareamento
  • Usado para: configuracao sem fio, treinamento de gestos, troca de voz

Ambas as APIs sao modernas e experimentais — funcionam nativamente no Chrome sem extensoes. O GESTUUM usa as duas em paralelo: USB para setup inicial, BLE para uso diario.

10. Treinamento de Gestos

O usuario treina novos gestos pelo web app. O processo exige 3 repeticoes do mesmo movimento:

1. App envia comando: {"cmd": "train_start", "gesture_id": "G01"} 2. Dispositivo entra em STATE_TRAINING 3. Usuario faz o gesto 3 vezes (com beep de confirmacao entre cada) 4. Para cada amostra: - Captura trajetoria (Matrix3D) + dados brutos (Orbital) - Armazena temporariamente 5. Apos 3 amostras: - Seleciona amostra mediana como referencia - Calcula threshold = media(distancias DTW entre amostras) x 1.5 - Extrai assinatura orbital (8 features) - Salva tudo no JSON do SPIFFS 6. Gesto marcado como "trained: true" — pronto para uso

Por que 3 repeticoes? Uma repeticao nao captura a variacao natural do movimento. Tres repeticoes permitem calcular um threshold realista — se as 3 amostras sao muito diferentes entre si, o threshold sobe (mais tolerante). Se sao quase identicas, o threshold desce (mais exigente).

12.2 Filosofia de Confiabilidade

Num dispositivo assistivo, um falso positivo (falar a palavra errada) e pior que um falso negativo (nao reconhecer). Por isso:

  • Ratio test de Lowe: Se dois gestos tem scores proximos (ratio > 0.95), rejeitar como ambiguo
  • Threshold por gesto: Cada gesto tem seu proprio limiar, calculado no treinamento
  • Score combinado: Exige concordancia de dois metodos (DTW 60% + Orbital 40%)
  • Modo emergencia: Threshold mais permissivo (4.5 vs 3.5) — em emergencia, e melhor falar algo do que nada

Zero falsos positivos > capturar todo gesto. Confiabilidade e dignidade.

14. Resumo para Apresentacao

O GESTUUM e um sistema de 2 microcontroladores ESP32 que se comunicam por ESP-NOW (latencia < 3 ms) para transformar gestos de mao em fala sintetizada. O reconhecimento usa matematica aplicada (grid 3D, PCA, DTW, correlacao) rodando localmente no chip a 240 MHz — sem internet, sem nuvem, sem dependencias.

O audio e pre-gerado por inteligencia artificial (Google Neural TTS) em 6 idiomas e 4 vozes, armazenado na memoria flash do dispositivo. A configuracao e feita por um aplicativo web que acessa o hardware diretamente pelo navegador (Web Serial + Web Bluetooth) — sem instalar nada.

O projeto usa 4 protocolos de comunicacao (ESP-NOW, BLE GATT, Bluetooth A2DP, USB Serial), 12 modulos de software independentes, e ~18.800 linhas de codigo — tudo open source.

A tecnologia e sofisticada, mas o objetivo e simples:

Dar voz a quem nao tem — com hardware de ~R$ 600, codigo aberto,
e tecnologia que cabe no pulso.
